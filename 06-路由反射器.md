# BGP 路由反射器（Route Reflector）

## 一、要解决的问题：iBGP 全互联

[03-BGP.md](03-BGP.md) 里讲过 iBGP 的一条铁律：**从一个 iBGP 邻居学到的路由，不会再通告给另一个 iBGP 邻居**（防环）。后果是一个 AS 内所有跑 BGP 的路由器必须两两建立 iBGP 会话，形成**全互联（full mesh）**，会话数按 n(n-1)/2 增长——10 台就要 45 条，100 台要近 5000 条，配置和资源都扛不住。

**路由反射器（RR）**就是来打破这条限制的：指定一台（或几台）路由器作为反射器，允许它把从一个 iBGP 邻居学到的路由"反射"给其他 iBGP 邻居，于是其余路由器只要和 RR 建一条会话即可，不用彼此全互联。

## 二、角色与术语

- **路由反射器（RR）**：负责反射路由的那台路由器。
- **客户机（Client）**：和 RR 建立会话、由 RR 为它反射路由的路由器。RR 上用 `route-reflector-client` 把某个邻居指定为客户机。
- **非客户机（Non-Client）**：AS 内既不是 RR 也不是某 RR 客户机的普通 iBGP 邻居。
- **簇（Cluster）**：一个 RR 和它的所有客户机组成一个簇，用 **Cluster-ID** 标识。

## 三、反射规则（核心）

RR 收到路由后按来源决定往哪反射：

| 路由来源 | 反射给客户机 | 反射给非客户机 | 反射给 eBGP 邻居 |
|---|---|---|---|
| 来自**客户机** | 是 | 是 | 是 |
| 来自**非客户机** | 是 | 否 | 是 |
| 来自 **eBGP** | 是 | 是 | 是 |

一句话记忆：**来自客户机的路由谁都给；来自非客户机的路由只给客户机**（因为非客户机之间本就要求全互联，已经直接收到了，不必重复反射）。

## 四、防环：两个新属性

打破了全互联，就需要新的防环手段。RR 在反射时会附加两个属性：

- **ORIGINATOR_ID**：记录这条路由在本 AS 内最初的始发者 Router ID。某路由器若收到 ORIGINATOR_ID 是自己的路由，直接丢弃。
- **CLUSTER_LIST**：路由每经过一个簇，就把该簇的 Cluster-ID 加进列表。RR 若发现列表里已有自己的 Cluster-ID，说明成环，丢弃。

## 五、配置实例

沿用 [04-IGP-BGP-结合.md](04-IGP-BGP-结合.md) 里 AS 65001 的 R1、R2、R3。原本三者要 iBGP 全互联（R1-R2、R2-R3、R1-R3 共三条）。现在把 **R2 设为 RR**，R1 和 R3 设为它的客户机——这样 R1 和 R3 之间就**不再需要直接建立 iBGP 会话**了。

**R2（路由反射器）**

```
router bgp 65001
 bgp router-id 2.2.2.2
 neighbor 1.1.1.1 remote-as 65001
 neighbor 1.1.1.1 update-source Loopback0
 neighbor 1.1.1.1 route-reflector-client      ! 把 R1 指定为客户机
 neighbor 3.3.3.3 remote-as 65001
 neighbor 3.3.3.3 update-source Loopback0
 neighbor 3.3.3.3 route-reflector-client      ! 把 R3 指定为客户机
```

**R1（客户机，只和 RR 建会话）**

```
router bgp 65001
 bgp router-id 1.1.1.1
 neighbor 2.2.2.2 remote-as 65001
 neighbor 2.2.2.2 update-source Loopback0
 neighbor 2.2.2.2 next-hop-self
 ! 注意：不再需要 neighbor 3.3.3.3 那条会话
```

**R3（客户机，只和 RR 建会话）**

```
router bgp 65001
 bgp router-id 3.3.3.3
 neighbor 2.2.2.2 remote-as 65001
 neighbor 2.2.2.2 update-source Loopback0
 neighbor 2.2.2.2 next-hop-self
```

R3 学到的外部路由发给 R2，R2 作为 RR 把它反射给客户机 R1；反向同理。R1 和 R3 互相的路由就这样通过 RR 打通了。

## 六、冗余设计

单台 RR 是单点故障。生产中通常在一个簇里放**两台 RR**，给它们配**相同的 Cluster-ID**，所有客户机同时和两台 RR 建会话：

```
router bgp 65001
 bgp cluster-id 1.1.1.1       ! 同一簇的两台 RR 配相同 cluster-id
```

配相同 Cluster-ID 的意义在于：两台 RR 互为对方反射的路由会因 CLUSTER_LIST 命中而被正确丢弃，避免簇内成环，同时又保留了双 RR 的冗余。

## 七、验证命令

```
show ip bgp summary                 # 确认客户机与 RR 的会话 Established
show ip bgp 172.16.5.0              # 看反射来的路由，含 Originator/Cluster 信息
show ip bgp neighbors 1.1.1.1       # 确认该邻居是否为 route-reflector-client
```

排错要点：客户机收不到反射路由时，先确认 RR 上确实加了 `route-reflector-client`（漏配会导致 RR 把客户机当普通 iBGP 邻居，从而不反射来自其他非客户机的路由）。

> 联盟（Confederation）是另一种解决 iBGP 全互联的方案，思路是把一个大 AS 拆成若干子 AS。实践中路由反射器用得远比联盟多，本仓库不再单独展开。
